1. Introduction
•
Dynamic programming: is a method for solving complex problems in an iterative or 
recursive manner, by breaking them down into simpler sub‐problems, such as solving 
the sub‐problems comes down to solving the initial problem
•
Applicable to problems having overlapping sub‐problems and optimal sub‐structure properties
• Overlapping sub‐problems: If the main problem can be broken down into sub‐problems 
• Optimal sub‐structure: The main problem’s optimal solution can be constructed efficiently 
from optimal solutions to its sub‐problems
• Dynamic programming is called divide and conquer techniques when the 
overlapping problems are much smaller than the original problem
•
Examples: 
•
Dijkstra’s OSPF algorithm: using in network routing and graph‐based applications  
•
Fibonacci sequence: using in mathematical‐based simulation problems
•
Sequence alignment and tree alignment (using edit distance)
•

2.1. Wagner & Fisher Approach
•
It is the cornerstone in string edit distance computations
–
Developed by R. A. Wagner and M. J. Fisher in 1974
– Simple algorithm using three basic edit operations: 
• Character insertion, noted: Ins(c) where c is a character
• Character deletion, noted: Del(c)
• Character update: noted Upd(c1, c2) where c1 and c2 are characters
– It takes as input: two strings of characters A and B
– It produces as output: a string edit distance value Dist(A, B)

2.2. Edit Operations
•
Formal edit operations syntax:
– Character insertion: Let c be a character to be inserted into a string S =S1, S2, …, 
Sm, Ins(c, i) is a character insertion operation applied to S, inserting c as the ith
character of S, transforming S into S’ = S’1, S’2, …, S’i‐1, c, S’i+1, …, Sm+1 
• Simplified syntax: Ins(c) will be used in the remainder to simplify presentation
– Character deletion: Let c be a character in S, Del(c) is a node deletion operation 
applied on S that removes c from S, producing S’= S’1, S’2, …, S’i‐1, S’i+1, …, Sm 
– Character update: Given a character c1 in S, and another character c2, Upd(c1, c2) is 
a character update operation applied to c1 resulting in a string S’ which is identical 
to S except that in S’, c1 is replaced with c2.

2.3. Edit Operations Costs
•
Different edit operation costs schemes can be used
– Intuitive cost model:
• CostIns(c) = CostDel(c) = 1
– Unit cost
• CostUpd(c1, c2) =         1 if c1 ≠ c2
0 otherwise
– NOTE: Different edit operation cost models can be chosen by the system designer
• Based on the application domain, given theoretical/experimental foundations
– For instance: we would consider the cost of updating a lower case letter into its upper 
case form to cost less than a normal update operation, etc.

2.7. Complexity Analysis
•
The time complexity of W&F’s algorithm is O(|A||B|)
– Where |A| and |B| designate the cardinalities (in number of characters) of the 
strings being compared
– Which amounts to O(N2)
• With N = max{|A|,|B|}
•
The space complexity of W&F’s algorithm is also O(|A||B|)
–
Which corresponds to the memory size required to store the edit distance matrix
•
The algorithm is of quadratic time & space complexity with respect to string size
–
It is linearly dependent on the size of each string

3.1. Document Tree Representation
•
Rooted Ordered Labeled Tree (ROLT): It is a rooted tree in which the nodes are labeled 
and ordered. We denote by T[i] the ith node of T in preorder traversal, T[i].its label, 
and T[i].d its depth. R(T)=T[0] designates the root node of tree T
•
Semi‐structured Document Tree: It is represented as a ROLT with a node corresponding 
to each element and attribute
–
Element nodes are labeled with element tag names
–
Attribute nodes appear as children of their encompassing element nodes, labeled and sorted 
by attribute name, and appearing before all sub‐element siblings
–
Element/attribute values can be considered or disregarded following the application at hand
• Values are disregarded when studying the structural properties of semi‐structured 
documents

3.2.4. Tree Edit Operations
•
Formal edit operations syntax:
– Leaf node insertion: Let T be a ROLT with a node p having first level sub‐trees P1, …, Pm
(i.e., sub‐trees rooted at the children nodes of p). Given a node x not belonging to T, 
Ins(x , p, i) is a node insertion applied to T, inserting x as the ith child of p, yielding T’ 
where p has first level sub‐trees P1, … , Pi‐1, x, Pi+1, … , Pm+1
• Simplified syntax: Ins(x) will be used in the remainder to simplify presentation
– Leaf node deletion: Given a leaf node x in ROLT T, x having parent node p with first level 
sub‐trees P1, …, Pm, Del(x) is a node deletion operation applied on T that removes x from 
T, yielding T’ where p has first level sub‐trees P1, … , Pi‐1, Pi+1, … , Pm
– Node update: Given a node x in ROLT T, and a label , Upd(x, ) is a node update 
operation applied to x resulting in T’ which is identical to T except that in T’, x bears as 
its label. The update operation could be also formulated as follows: Upd(x, y) where y. 
denotes the new label to be assumed by x

3.2.9. Complexity Analysis
•
The time complexity of Chawathe’s algorithm is O(|A| + |B|+|A||B|) = O(|A||B|) 
–
Where O(|A| + |B|) designates the complexity of producing LD‐pair strings
–
|A| and |B| designate the cardinalities (in number of nodes) of the trees being compared, 
which are equivalent to the cardinalities of LD‐pair strings |A*| and |B*| respectively
– Hence, complexity simplifies to O(N2)
• With N = max{|A|,|B|}
•
The space complexity of Chawathe’s algorithm is also O(|A||B|)
–
Which corresponds to the memory size required to store the edit distance matrix
•
The algorithm is of quadratic complexity with respect to tree size
–
It is linearly dependent on the size of each tree

Which corresponds to the memory size required to store all edit distance matrixes

The algorithm is of quadratic complexity with respect to tree size

4. Applications of TED‐based Similarity
4.4. Data Integration
•
Data integration task consists of two main steps:
1. Similarity evaluation: performing an approximate join between data
• Given two data sources, S1 and S2, a similarity threshold ThreshSim, and a similarity function 
Sim(d1, d2) between two documents d1 (from S1) and d2 (from S2)
• Output mapping between all pairs of documents (d1, d2) such that Sim(d1, d2) ThreshSim
2. Data merging: Paired documents are merged together to form unified views
• Differencing (TED) techniques are particularly useful here
– Processing the edit script (diff/delta) to identify commonalities and differences 
between the paired documents
– These provide a mapping between document trees used to perform merging
‐ Given predefined merging rules (e.g., given two identical nodes, eliminate one 
and preserve the other)

4.5. Approximate Querying
•
Efficient retrieval of semi‐structured documents becomes more and more important
–
With the increasing use of semi‐structured data, specifically on the Web
•
The DB community has proposed several languages for querying semi‐structured data
–
Including XML‐QL, XQL and XQuery
Based on exact matching and do not support ranked queries
•
Several attempts have been made to extend these query languages, using similarity‐
based processing, in order to allow ranked results
1.
Comparing query data (documents/fragments/elements) with  document collection
2.
Returning data which similarity with query data is above user chosen ThreshSim
3.
Results are ranked based on similarity
