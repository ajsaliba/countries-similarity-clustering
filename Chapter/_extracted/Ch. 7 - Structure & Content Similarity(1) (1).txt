1. Introduction
•
Semi-structured data similarity is one of the central topics in the Information Retrieval 
(IR) field
•
While TED (Tree Edit Distance)-based  approaches for comparing semi-structured data:
– Are designed to handle rigorously structured (data-centric) data
– Are fine-grained, and thus require sophisticated processing and time to execute
– Main target: data structure
– Applications: Data warehousing, data integration, and structural 
clustering/classification
•
IR-based methods for comparing semi-structured data:
– Are designed to handle loosely structured (document-centric) data
– Are coarse-grained, and thus require less processing time to execute
– Mainly target: data structure and content
– Application: ranked querying

2. Traditional IR Techniques
2.3. Term Indexing
•
The TF (Term Frequency) factor designates the number of times a term ti 
occurs in document D (document statistics)
– The underlying idea is that the importance of a given term ti in describing a 
document D increases with the frequent use of ti in D
•
The IDF (Inverse Document Frequency) factor indicates the fraction of 
documents that contain term ti (collection statistics)
• The underlying idea is that the importance of a given term ti in describing a 
document D decreases with the frequent use of ti in the document collection C
– A term appearing in more documents becomes less distinctive/descriptive of a 
specific document
   IDF estimates the rarity of ti in the document collection C

•
Every DB-style query exactly defines a set of rows to be retrieved
– Every row is an exact/complete match of the query: and thus a correct answer 
 
  There is never a bad hit for a DB-style query
•
An IR-style query can return bad hits
– Documents in the hit list only approximately/partially match the query 
• The query does not provide a complete specification of the user's intention
– Every query has an implied intention that cannot be described fully in the 
query and hence is not available to the retrieval engine

2. Traditional IR Techniques
2.6. Evaluation
•
Need for IR evaluation measures
– Based on the idea of benefit/cost of a retrieved document to the user
• A good (relevant) hit has a certain benefit to the user
• A bad (non-relevant) hit has a certain cost to the user: time spent reading it, cost of 
getting the document translated, edited, and so on
•
Main measures used:
– Precision: percentage of relevant documents correctly retrieved by the system 
with respect to all documents retrieved by the system
– Recall:  percentage of relevant documents correctly retrieved by the system with 
respect to all documents relevant for the user

3.1. Indexing Nodes
•
A semi-structured document is generally processed as a number of indexing 
nodes [9]
– Atomic units in the document, i.e., elements (attributes), which encompass disjoint 
sub-trees
• Each indexing node underlines a distinct indexing context
– Such that TF-IDF weights can be computed locally, instead of being 
evaluated w.r.t. to the whole document
– The weights would then be augmented, down-weighting the TF-IDF scores 
when the terms are propagated upwards in the document tree
 Thus taking into account the hierarchical aspect of semi-structured data

Processing is done at the document level

3.5. Matrix Model
•
Extending/transforming term/context vector representation model
– To matrix representation model [12]
•
In the vector space model, a weight wD(ti) is expressed by a real number (e.g. TF-IDF 
score) specifying the distribution of a term ti for the entire document
– In the matrix model, the weight of a term is expressed by a vector 
• Reflecting the distribution of the term ti in the document paths
– W.r.t. to each path occurring in the document collection

4. Underlying Indexing Techniques
•
Processing keyword-based queries is a fundamental problem in IR 
– Main premise of keyword-based querying:
1. The user issues a keyword query containing a bunch of terms
2. Then, documents containing the query terms are retrieved
• Ranked following their relevance (similarity) w.r.t. the query
•
The brute force approach is to sequentially scan documents
– Searching for terms that map to the query keywords
• Problem: Very inefficient for big document collections
•
Solution: Use some sort of indexing to speed up the search process

5. From IR to AI-enabled RAG
•
Traditional Information Retrieval (IR)
– Indexing, ranking, querying using VSM, TF-IDF, context paths, etc.
– Returns documents that are relevant w.r.t. a user query
•
Large Language Models (LLMs)
– Provide fluent natural language answers
 
– But rely on learned, static internal parameters
 
 
→ Prone to hallucinations, outdated knowledge 
•
Motivation for Retrieval Augmented Generation (RAG)
– Combine IR precision + LLM generation
– Ensure answers are grounded in retrieved and relevant content
– Leverage semi-structured data, structured metadata, or raw text

5. From IR to AI-enabled RAG
RAG Pipeline Overview
1. Query Processing
•
User issues a natural-language question
•
The system encodes it into a vector representation (similar to VSM)
2. Retrieval Component (IR)
•
Searches in an inverted index or semi-structured index
•
Uses similarity search (cosine / ANN / semantic similarity)
•
Retrieves top-k relevant chunks / nodes / passages
3. Augmentation Step
•
Retrieved content is injected into the LLM prompt
–
Functions similarly to context weighting, category restriction, or term-in-context indexing
4. Generation
•
LLM generates an answer grounded in the retrieved evidence
–
Reduces hallucination; increases domain-specific accuracy
